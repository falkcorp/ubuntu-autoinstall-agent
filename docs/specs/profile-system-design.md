## Composable Component Profiles — Design Spec (rev. 2)

### 1. Problem

Every provisioning decision for a host lives in one flat `InstallationConfig` struct (`crates/uaa-core/src/network/ssh_installer/config.rs:78-173`). Disk geometry, encryption unlock factors, network, base image, workloads, and firmware workarounds are interleaved as ~25 sibling fields, plus behavior that is not config at all — firmware quirks sniffed from a device-path string (`disk_device.starts_with("/dev/md")`), NIC drivers live-probed, serial-console applied unconditionally (`system_setup.rs:671`), RESET staging applied unconditionally, and CockroachDB cluster membership baked into the `LENSERV_MEMBER_IPS` constant (`applications.rs:26`). There is no hook mechanism, so every host-specific behavior becomes a new hardcoded installer branch — the "rebuild a one-off per change" pattern this design ends.

A partial profile system already exists and works: `HostGroupProfile` (defaults) + `HostProfile` (overrides) resolve through `merge()` into `InstallationConfig`, recording per-field `Provenance`, persisted in `uaa-control`'s registry and served by `uaa config place --from-registry`. **This design evolves that system into composable components — it does not replace it.**

### 2. Goals & non-goals

**Goals.** Decompose the monolith into independently-selectable COMPONENTS (disk-layout, unlock-policy, network, base-image, firmware-quirks, hooks, applications) plus two CLASSIFIERS (arch, role). Give operators a compose-from-components authoring surface. Keep the len-serv PlainLuks placed artifact **byte-identical** until explicitly migrated. Stage the migration reversibly and rollback-safely. **Express** U1 (native-keystore/clevis), len-serv (single-luks/dracut/cockroach), and rpi/tang (arm64 + tang-server role) through one authoring+validation interface — with the honest qualifier that rpi coverage is *expressibility + validation*, not a proven-bootable arm64 install (§9, §11).

**Non-goals.** No greenfield schema. No third merge tier and **no component-bundle/desugar tier** (cut — §4). No parallel "resolved components" type-space (§3). No plugin framework (components are closed-but-growing enums). Not a promise to boot arm64 tomorrow (§9).

### 3. The component model — one wire type, authoring-side components

A **component** is one axis of provisioning decision. The key structural decision, revised for simplicity: **there is exactly one resolved/wire type — `InstallationConfig`, its flat-field ABI preserved.** Components are an *authoring-side* concern: nested sub-structs on `InstallationConfigPartial`. There is **no** parallel `ResolvedComponents` type. `lower()` is the single bridge that maps authoring nesting onto the flat wire fields.

Two structural shapes, borrowed from code that already exists:

- **Variant-select components** (disk-layout, applications, firmware-quirks) are closed tagged enums — the exact `ApplicationSpec` shape (`tag="kind"`, `deny_unknown_fields`). Selecting a variant is **whole-replace**. These three (and only these) carry the lowering seam plus a round-trip equality gate.
- **Field components** (unlock-policy, network, base-image) are plain nested authoring sub-structs whose fields merge independently host-over-group and **field-copy** into existing flat wire fields. No variant machinery, no separate resolved type.

| Component | Kind | Variants / factors | Wire vehicle |
|---|---|---|---|
| arch (classifier) | scalar enum (NEW, in `ssh_installer::config`) | amd64\*, arm64 | additive skip-if-amd64 |
| role (classifier) | scalar enum | install-target\*, tang-server | additive skip-if-default |
| disk-layout | variant-select | single-luks, zfs-native-keystore | lowers to `disk_device`/`storage_mode`/`disks` |
| unlock-policy | field | tang-sss, tpm2-pin, tpm2-clevis-peer, fido2 | lowers to the 6 flat unlock fields |
| network | field | static / dhcp addressing | lowers to the 6 flat network fields |
| base-image | field | release, mirror, initramfs, fallback | lowers to `debootstrap_*`/`initramfs_type` |
| firmware-quirks | variant-select (Vec) | grub-removable, force-nic-driver, watchdog-staggered | NEW additive skip-if-empty |
| hooks | keyed step list | pre/post per **new** `Phase` enum | NEW additive skip-if-empty |
| applications | variant-select | cockroach, tang-server (new) | already present |

\* = default; a config that omits the component serializes exactly as today.

**Serial-console and nvme-cant-boot are deliberately NOT firmware-quirk variants** — see §11.

### 4. Merge semantics (the crux)

`merge.rs` stays **two-tier**: group defaults → host overrides. Each component threads through the same passes:

1. **Variant SELECT = whole-replace by kind.** Generalizes `union_applications` (`merge.rs:167-188`): a kind present in one tier is taken as-is; present in both, the host's whole entry wins.
2. **Field PARTIAL**, both (a) *within* a selected variant (`merge_cockroach`, `merge.rs:196-222`, spec D1) and (b) for field components, each factor merging host-over-group independently.

**`unlock-policy` is the correctness canary.** Its `tpm2_pin` factor keeps the double-`Option<Option<String>>` inherit-vs-explicit-none distinction (`profile/mod.rs:65-70`): a host meant to have NO pin must not silently inherit the group's. `resolve_double_option` (`merge.rs:104-120`) is reused unchanged; `test_tpm2_pin_explicit_none_does_not_inherit` must still pass **through `raise → merge → lower()`**.

**Provenance deepens, never drops.** The `Provenance` map (relied on by DS-OPS drift review) becomes keyed by component authoring-path — `unlock-policy.tang.threshold → Source::Group`. Drift review folds in unchanged because it diffs the resolved blob's `content_hash`.

**No bundle tier.** The prior "profile = named component bundle that desugars into group defaults" concept is **cut from this brief**. At ~2-3 groups a shared component set is plain copy-paste across two group YAMLs; a bundle-expansion pass adds resolution-order reasoning for a single candidate use. Revisit only after copy-paste causes a real drift incident.

### 5. The lowering seam

Resolution stays: `resolve_from_registry` (`profiles/resolve.rs:46`) → `merge()`. A new **pure, total** `lower(resolved InstallationConfigPartial) → InstallationConfig` runs immediately after `merge`, **on the control plane**. It is modeled on `layout.rs::plan_layout` — the existing pure-planner→applier split.

```
registry rows ─► merge() ─► resolved partial ─► lower() ─► InstallationConfig ─► config_place ─► served YAML
   (2-tier, provenance)     (typed authoring)   (pure bridge)  (unchanged flat ABI)  (placeholders)   (wire artifact)
```

Because every component lowers into existing/additive flat wire fields, **`lower()` is the only new type-bridging code** — no parallel resolved type to keep in sync. The byte-identical guarantee (DS-OPS-03) is about the *placed/served wire artifact* an older `uaa install` binary must still parse; keeping it = lowered `InstallationConfig` preserves that. `config_place`'s `REPLACE_AT_PLACE_TIME` hard gate operates on the lowered struct exactly as today — guarded by a new **placeholder-survival test** per secret-bearing field (§9).

### 6. Composition legality — a NEW `validate_resolved()`

`profile/validate.rs` is **not a stub**: it is a working 439-line module (`validate(groups, profiles)` at `:41`, plus `check_global_hostname_uniqueness` / `check_prefix_uniqueness` / `check_standalone_rules` / `check_hostname_pattern` / `check_no_rename`) that runs **pre-merge** over the whole group/profile list. Composition legality is a property of a single **post-merge** `InstallationConfig` and does not fit that signature. Add a distinct sibling:

```rust
pub fn validate_resolved(cfg: &InstallationConfig) -> Result<()>
```

Rules (this is where arch/role classifiers do their work):
- `native-keystore` requires a non-empty `disks` roster and `arch = amd64`.
- a `tang-sss` factor requires non-empty servers with `threshold ≤ len`.
- `tpm2-clevis-peer` is legal only with `disk-layout = native-keystore` (encodes today's hidden `include_tpm2_peer` coupling explicitly).
- `arm64` forbids `grub-removable-fallback` (x86-only).
- `role = tang-server` permits empty disk/unlock but requires a `tang-server` application; `install-target` requires disk-layout + unlock-policy.
- (optional) cross-check that a `NativeKeystore` roster does not assign `DiskRole::System` to an NVMe device on an NVMe-can't-boot board — the home for the U1 constraint, replacing a would-be standalone quirk.

### 7. The three hosts in one interface

```yaml
# len-serv (group)  — single-luks / dracut / cockroach
arch: amd64
role: install-target
disk-layout:   { kind: single-luks, disk-device: /dev/nvme0n1 }
unlock-policy: { tang: { servers: [...45,...46,...47], threshold: 2 },
                 tpm2-pin: { pin: REPLACE_AT_PLACE_TIME, pcr-ids: "7" },
                 fido2-expected: true }
base-image:    { release: resolute, initramfs: dracut }
applications:  [ { kind: cockroach } ]        # join derived from group allocations (Phase 1)
# serial-console is arch-implied, injected by lower(); it is NOT in any quirks vec

# unimatrixone (standalone group) — native-keystore / clevis D2-B
arch: amd64
role: install-target
disk-layout:   { kind: zfs-native-keystore, disks: [ {id: ...,role: system} x2, {id: ...,role: special} x2 ] }
unlock-policy: { tang: { servers: [...], threshold: 2 }, tpm2-clevis-peer: true }  # enroll_tpm2 = false
# nvme-can't-boot lives in DiskRole::System, enforced by validate_resolved()

# rpi-serv (group) — arm64 / tang-server role  (EXPRESSIBILITY + validation only)
arch: arm64
role: tang-server
base-image:    { release: resolute, initramfs: initramfs-tools }   # arm64 mirror
applications:  [ { kind: tang-server } ]
firmware-quirks: [ watchdog-staggered ]   # greenfield shape — needs operator requirements
```

The same three components (arch, role, applications) plus optional payload components express all three classes **at the authoring/validation layer**. rpi gains representation the current `InstallationConfig` entirely lacks. A bootable arm64 install is a separate, unfunded piece of work (§9).

### 8. Operator interface

Compose a host by naming component variants; the resolver lowers and the installer consumes. Operators interact through the existing `/api/profiles` + `/api/groups` routes (DS-OPS-01) and `uaa config place --from-registry` — both already resolve through `merge`, so they gain component authoring without new surface. A `uaa profile explain <host>` view surfaces the component-path `Provenance`.

### 9. Migration — staged, reversible, rollback-safe, byte-identical-gated

The two invariants of the prior plan (additive `skip_serializing_if` fields; total `lower()`) are **necessary but not sufficient**. They protect the *placed YAML*, but **not** the *registry authoring blob*: `InstallationConfigPartial` is `#[serde(deny_unknown_fields)]` (`mod.rs:40`), and `convert.rs` deserializes stored JSON into it with `resolve_from_registry` failing **group-scoped** on any parse error. A migrated **shared** group blob (len-serv = all 3 CockroachDB nodes) would fail-close fleet-wide under an older binary. The `skip_serializing_if` trick does not transfer — a migrated host's component key is present by definition. This drives the phase ordering below.

| Phase | Goal | Reversible |
|---|---|---|
| 0 | Land component authoring types + new `Phase` enum + new `Arch` enum + `lower()` + `validate_resolved()` behind an unused seam. Gate = compare `lower()` of a **hand-authored** component profile for each committed host against its committed/placed config (NOT a `lower(raise())` identity test); merge-blocking CI. | delete unused types |
| 1 | Add arch/role/firmware-quirks/hooks as additive skip-if-empty fields; defaults reproduce today. Serial-console becomes an arch-implied `lower()` injection (never in a serialized vec). Remove `/dev/md` IMSM sniffing (IMSM being dropped). **Early payoff:** derive cockroach membership from the group allocation roster, retiring `LENSERV_MEMBER_IPS`, gated by `derived == constant`. | skip-if-empty fields; reinstate constant |
| 2 | **Rollback-safety prerequisite.** Deploy the component-aware **control** binary fleet-wide + a stored `schema_version` on the row the control binary refuses to serve/roll-back below. Expand step of expand-then-migrate; migrates zero blobs. | roll-forward superset |
| 3 | Migrate **U1** (standalone single-host group — rollback blast radius = 1 host). Re-pass the D2-B VM gate; placeholder-survival test for keystore secrets. | revert U1 profile |
| 4 | Author the **rpi-serv** group. **Explicit scope: expressibility + `validate_resolved()` coverage only — NOT a bootable arm64 install** (needs an x86-assumption audit, below). Isolated net-new group. | delete rpi group |
| 5 | Migrate the **shared len-serv** group one component at a time (network → base-image → unlock-policy → disk-layout). Runs only after Phase 2. Double-gated per step: (a) hand-authored-`lower()` == committed equality (merge-blocking on that PR) and (b) placeholder-survival for any secret field. Documented rule: *no control rollback below `schema_version X` once len-serv defaults are migrated.* | per-component revert (content); version gate covers binary rollback |
| 6 | Installer modules consume typed resolved fields (formalizing the `struct{runner}+async fn(&config)` trait); retire legacy `autoinstall/render.rs` template and dead `image/deployer.rs`+`customizer.rs`+`TargetConfig` (which owns the legacy `Architecture` enum). Flip only after the whole fleet's *installer* binary is component-aware. | per-module revert to flat field |

**x86-assumption audit (blocks a bootable rpi install, not this brief):** dracut/GRUB/tpm2 chroot commands throughout `system_setup.rs` are x86-oriented; Phase 4 does not resolve them.

### 10. What this reuses vs. adds

**Reuses unchanged:** `merge.rs` resolvers + fail-closed 10-field set + `resolve_double_option`; `union_applications`/`merge_cockroach` as the two merge primitives; `ApplicationSpec` closed-enum pattern; `StorageMode`/`DiskSpec`/`DiskRole` as both disk-layout seed and its lowering target; `InitramfsType::regenerate_cmd()`; `resolve_from_registry`; `config_place` placeholder gate; `layout.rs::plan_layout` as the lowering model. **No DB migration** — `HostGroupRow.defaults`/`applications` are `serde_json::Value`.

**Adds:** the authoring sub-structs + per-variant partials; a **new** `Arch` enum in `ssh_installer::config` (not the legacy `config::Architecture`); a **new** `Phase` enum (not `PhaseSelection`, which is a bool-array struct); `lower()`; a **new** `validate_resolved(&InstallationConfig)` sibling; four additive axes; a `tang-server` `ApplicationSpec` variant; a `schema_version` row gate; component-path provenance; per-component hand-authored-equality and placeholder-survival tests.

### 11. Explicit decisions (deltas from rev. 1, all judge-driven)

1. **Serial-console** is an arch-implied default `lower()` injects unconditionally for amd64 install-targets — it never appears in a host's serialized `firmware_quirks` vec, so the empty-vec byte-identical claim holds. (Resolves the rev-1 self-contradiction.)
2. **nvme-cant-boot** is **dropped** as a quirk; the constraint stays in `DiskRole::System`'s semantics (`config.rs:203-205`), optionally enforced by `validate_resolved()`. No second source of truth.
3. **hooks** uses a **new** `Phase` enum (7 variants ↔ the `run_phase!` labels at `installer.rs:295-342`), described as new scaffolding — `PhaseSelection` (`installer.rs:41`) is a private-field bool-array struct and cannot key a `BTreeMap`.
4. **arch** is a **new** enum in `ssh_installer::config`; the legacy `config::Architecture` belongs to the retired `TargetConfig`/image pipeline.
5. **validate.rs** is *extended* with `validate_resolved()`; it is a working 439-line pre-merge module, not a stub.
6. **Rollback safety** at the authoring layer via **expand-then-migrate + a `schema_version` gate**, shared len-serv sequenced last. `deny_unknown_fields` is **kept** (real typo detection) — the judge's blanket-removal option is rejected in favor of expand-then-migrate.
7. **M2 gate** compares `lower()` of a hand-authored profile vs committed config, not a tautological round-trip; merge-blocking per PR.
8. **Placeholder-survival tests** are required per secret-bearing component before its phase ships.
9. **Type-space simplification:** only disk-layout/applications/firmware-quirks are variant-select with a lowering gate; network/base-image/unlock-policy are plain authoring sub-structs that field-copy into existing flat wire fields — one wire type, no parallel `ResolvedComponents`.
10. **Bundle/desugar tier cut** from this brief.
11. **Early payoff:** cockroach-membership derivation (retiring `LENSERV_MEMBER_IPS`) lands in Phase 1, so the model deletes a live one-off before the full migration; Phases 3-4 are honestly scoped as expressibility/validation.
12. Legacy `autoinstall/render.rs` template **retired** after Phase 5; dead `image/deployer.rs`+`TargetConfig` **removed** (it owns the legacy `Architecture`). IMSM/mdraid quirk **not modeled** (IMSM being dropped for native ZFS).

## Appendix B — Layering & mapping onto the existing registry

## Appendix C — Settled decisions

- **Serial-console is an ARCH-IMPLIED default injected by lower(), NOT a per-host FirmwareQuirk entry.** — configure_serial_console() runs unconditionally in every install today (system_setup.rs:671, SERIAL_CONSOLE_DROPIN at :32). The prior design self-contradicted: it listed serial-console as a firmware_quirks variant 'for all amd64 servers' AND claimed firmware_quirks stays empty for byte-identical serialization. Resolving the fork by horn (a): lower() injects serial-console unconditionally for amd64 install-targets, so it NEVER appears in a host's serialized firmware_quirks vec — the empty-vec skip_serializing_if byte-identical claim (proven by test_plain_luks_host_omits_storage_keys) holds, and there is no free 'defaults reproduce today' hand-wave hiding a shape change.
- **nvme-cant-boot is DROPPED as a firmware-quirk; the constraint stays in DiskRole::System semantics, optionally enforced by a validate_resolved() cross-check.** — DiskRole::System's doc comment (config.rs:203-205) already encodes 'X10DSC+ firmware cannot boot from NVMe, so the ESP/bootloader must live here', and layout.rs::plan_layout already consumes the roster to place ESP/bpool. A separate no-param NvmeCantBoot quirk would be either inert duplication or a second disconnected source of truth. If enforcement is wanted, add a validate_resolved() rule that cross-checks the NativeKeystore roster's DiskRole::System assignment against a board's NVMe-boot capability — not a standalone quirk with its own enable/disable semantics.
- **hooks uses a NEW enum Phase (7 variants mapped 1:1 to the run_phase! labels), described as new scaffolding — NOT reused.** — The prior design cited 'the existing PhaseSelection enum'. Verified: PhaseSelection (installer.rs:41) is a struct { selected:[bool;7], explicit:bool } with private fields — not an enum, not Ord/Hash, cannot key a BTreeMap; no Phase type exists anywhere. A mechanical implementer following the old citation could not compile it. The new Phase enum is cheap and correct, but the mapping_to_existing claim must not count it as free reuse.
- **arch is a NEW enum in ssh_installer::config, NOT the existing crate::config::Architecture.** — crate::config::Architecture (config/mod.rs:20) is consumed exclusively by the TargetConfig/image-builder pipeline this design's own Decisions retire. Building load-bearing profile machinery on a type slated for deletion is a self-inflicted contradiction. A new Arch enum in the ssh_installer config space keeps the profile system independent of the retirement.
- **validate.rs is EXTENDED with a new validate_resolved(&InstallationConfig) sibling; it is not a stub being filled.** — validate.rs is a working 439-line module operating PRE-MERGE over group/profile slices (naming/uniqueness invariants). Composition legality is a property of a single POST-MERGE InstallationConfig and does not fit the validate(groups,profiles) signature. A distinct post-merge entry point is the correct home; describing it as 'filling the DS-PRF-03 stub' would misdirect a mechanical implementer.
- **Rollback safety at the REGISTRY-AUTHORING layer via expand-then-migrate + a schema_version gate — NOT by removing deny_unknown_fields.** — InstallationConfigPartial is deny_unknown_fields (mod.rs:40) and convert.rs deserializes stored JSON into it, with resolve_from_registry failing GROUP-scoped on parse error. A migrated SHARED group blob (len-serv, all 3 CockroachDB nodes) would fail-close fleet-wide under an older binary. The prior design's skip_serializing_if trick does NOT transfer — a migrated host's component key is present and non-default by definition. Fix: (Phase 2) deploy the component-aware control binary fleet-wide BEFORE any blob gains a component key, plus a stored schema_version the control binary refuses to serve below, plus a documented no-rollback-below-X rule with the shared len-serv group sequenced LAST. I REJECT the judge's option of blanket-removing deny_unknown_fields — it is real typo-detection on authored profiles; expand-then-migrate achieves cross-version safety without weakening it.
- **The M2 byte-identical gate compares lower() of a HAND-AUTHORED component profile against committed/placed config — not a round-trip identity test.** — lower(raise(committed))==committed is tautological and gives false confidence for the exact Phase-5 hazard: an independently hand-authored registry component profile whose lowered geometry constants drift from disk_ops.rs's 512M/4G/2G literals. The gate must exercise the real authoring path and be merge-blocking on each migrating PR.
- **A placeholder-survival test is required per secret-bearing component before its migration ships.** — config_place's REPLACE_AT_PLACE_TIME hard gate scans the flat InstallationConfig. Once tpm2_pin/luks_key move into nested authoring sub-structs traversing an extra merge->lower hop, nothing proves the placeholder threads through unchanged. An explicit raise->merge->lower placeholder-survival test per field (mirroring test_tpm2_pin_explicit_none_does_not_inherit) keeps the hard gate protecting migrated hosts.
- **Only disk-layout, applications, and firmware-quirks carry the lowering seam + round-trip gate; network/base-image/unlock-policy are plain nested authoring sub-structs that field-copy into existing flat wire fields.** — The simplicity lens is right that a full component+partial+separate-resolved-type for non-variant data doubles the sync surface. Refinement of the judge's literal suggestion: rather than adding nested sub-structs to the WIRE type InstallationConfig (which already has the flat fields — that would duplicate on the ABI), keep components AUTHORING-side only on InstallationConfigPartial and preserve ONE wire type with its flat ABI. Result: no parallel ResolvedComponents type at all; lower() is the only bridge. This satisfies the judge's goal more cleanly than nesting on the wire type would.
- **The 'profile = named component bundle / desugar' middle tier is CUT from this brief.** — Not in the user's ask; at ~2-3 groups a shared serial-console+base-image set is plain copy-paste. A bundle-expansion pass adds resolution-order reasoning (group-authored vs bundle-authored, expansion before/after override) for a single candidate use. Defer until copy-paste causes a real drift incident.
- **'Expresses U1 + len-serv + rpi' is qualified everywhere: rpi is authoring/validation EXPRESSIBILITY, not a bootable arm64 install.** — Verified zero tang-server/watchdog/arm64-provisioning code exists in the repo. Phase 4 delivers a valid rpi group + validate_resolved() coverage; a bootable arm64 tang-server install needs a per-phase x86-assumption audit (dracut/GRUB/tpm2 chroot commands). The qualifier travels with every coverage claim, not just the phase table.
- **Payoff is proven early, not entirely deferred: cockroach membership derivation lands in Phase 1.** — The prior plan gated all rebuild-avoidance behind the highest-risk final phase. Pulling the LENSERV_MEMBER_IPS retirement (a live one-off hardcode, applications.rs:26) into the first working phase — guarded by 'derived==constant for current fleet' — proves the component model deletes real hardcodes before the full migration, and honestly scopes Phases 3-4 as expressibility/validation.
- **IMSM/mdraid quirk deliberately NOT modeled.** — Per settled architecture (memory: Storage Arch: Drop IMSM for Native ZFS), IMSM is being dropped for native ZFS on U1; a declarative md-assemble quirk would model a path being deleted. The /dev/md sniffing is removed in Phase 1, not componentized.
